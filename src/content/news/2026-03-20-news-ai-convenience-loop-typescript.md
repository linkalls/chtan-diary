## TypeScript首位って、ただのフロントエンド慣性じゃなかった

GitHubまわりの最新データを見ていると、ちょっと面白いというか、かなり露骨な変化が起きている。**TypeScriptがGitHub上で最も使われる言語になった**という話自体は「まあそうだよね」で流しそうになるんだけど、今回の本題はそこじゃない。もっとヤバいのは、**AIコーディング支援が“どの言語を選ぶか”そのものを押し始めている**ことです。

昔の技術選定って、性能、ライブラリ、採用しやすさ、チームの習熟度、そのへんの総合格闘技だった。でもいまはそこに新しい軸が1本増えた。**その言語、AIがどれだけ気持ちよく書ける？** っていう軸だ。しかもこれ、補助的な要素じゃなくて、開発体験そのものに直結するから強い。人間は便利さに弱い。開発者も例外じゃない。むしろかなり弱い。

## GitHubが言い出した「convenience loop」、名前は軽いのに中身は重い

GitHubはこの現象を **convenience loop** と呼んでいる。ざっくり言うと、AIでラクに書ける技術ほど選ばれやすくなり、選ばれるほどコード例が増え、コード例が増えるほどAIがさらに上手に書けるようになる、という自己強化ループだ。要するに、**使いやすいから普及する**じゃなくて、**AIが書きやすいから、ますます使いやすく見える**構造になってきた。

この話、ふわっとした感想戦じゃないのが怖い。GitHubの Octoverse 2025 では、2025年8月に TypeScript が Python と JavaScript を抜いて首位に立った。年次成長率も強いし、新規開発者のかなり早い段階で Copilot が当たり前に使われている。しかも公開リポジトリで LLM SDK を使うプロジェクトが **110万超** に達している。つまり、AI支援開発はもう一部の先進チームの遊びじゃなくて、**開発環境の地形そのもの** になっている。

### いま起きている変化を雑にまとめるとこんな感じ

- TypeScript が GitHub 上で最頻出級の言語になった
- 新規開発者の多くが最初の週から Copilot を使う
- LLM SDK を使う公開リポジトリは 110 万超
- 「書きやすい言語」より「AIが補助しやすい言語」が強くなる
- 技術選定が人間の好みだけでなく、AIとの相性で歪み始めている

この並びを見ると、もう「AIで少し速くなりました」程度の話じゃない。**AIがエコシステムの重心をずらし始めている**。

## なぜTypeScriptがAIと相性いいのか。答えはロマンじゃなくて型

ここで TypeScript が強い理由は、思想がクールだからでも、フロントエンド界の覇権だからでもない。かなり地味で、かなり本質的で、やっぱり**型**です。型があると、AIが「やっていいこと」と「やると壊れること」の境界を把握しやすい。`string` に対して数値演算を生やしたり、存在しないプロパティを当然のように触ったりする雑な事故を、少なくとも早い段階で止めやすい。

JavaScript は自由だし、その自由さが魅力でもある。でもAIにとっての自由は、しばしば**霧**になる。何でもできる環境では、正解の候補もノイズも両方増える。対して TypeScript は、書けるコードの空間をある程度狭める。その狭さが人間には窮屈でも、AIにはむしろありがたい。乱暴に言うと、**TypeScript は AI にとって親切なガードレールつき道路**なんですよね。

## いちばん割を食うのは「新しいけど学習データが薄い言語」かもしれない

この流れで地味にしんどいのが、新しい言語やニッチな言語です。言語そのものが優れていても、AIが十分な量のコード例を見ていなければ補助が弱い。補助が弱いと開発者は選びづらい。選ばれないとコードが増えない。するとまたAIが苦手なまま残る。これ、完全に**勝者がさらに勝つループ**になってる。

たとえば「設計は好き」「思想も美しい」「でも Copilot や各種エージェントが微妙に頼れない」という言語は、2026年の現場だとその時点でかなり不利になりうる。以前は「コミュニティが小さいけど将来性あるね」で済んだ話が、いまは **AI支援の弱さが日々の摩擦として可視化される**。ここがエグい。技術の優劣だけじゃなく、**AIにどれだけ“理解されているか”が採用コストに直結する**時代に入ってる。

## これ、TypeScript礼賛で終わらせるとむしろ危ない

ただし、この話を「やっぱ TypeScript 最強」で締めるのは雑すぎる。大事なのは、**AIが強い技術を選ぶほど、その技術にロックインされやすくなる**ことです。便利だから採用する。便利だから教育コストも下がる。便利だから生成量も増える。でも便利さに流されると、アーキテクチャ判断まで“AIが気持ちよく補完できるか”に寄りすぎる危険もある。

本来、技術選定って「保守性」「業務要件」「チーム構成」「運用」「将来の変更耐性」まで含めて決めるものです。そこに AI 相性が加わるのは自然だけど、**AI相性が最上位の評価軸に化けると、判断がかなり短期最適化しやすい**。言い換えると、便利さは武器だけど、便利さは麻薬でもある。

### じゃあどう見るべきか

- AI が強い言語を使うのは合理的
- でも「AI が得意」だけで技術選定しない
- 型、テスト、レビュー、ADR みたいな人間側の設計資産はむしろ重要になる
- 新しい言語や小さい言語は、AI支援込みで導入コストを見積もる必要がある
- 開発体験の差は、今後ますます採用競争そのものを変える

## 2026年の技術選定、もう“言語戦争”じゃなく“AIとの共生戦争”だ

今回のGitHubデータでいちばん面白いのは、AIがコードを書く量を増やしたことじゃない。**AIが開発者の好みを再配線し始めた**ことだ。TypeScriptの上昇は、単なる一時的ブームというより、AI時代の技術選定がどこへ向かうかを先に見せている気がする。

たぶんこれからの勝ち筋は、「最速の言語」でも「いちばん美しい言語」でもなく、**人間が設計しやすく、AIが補助しやすく、運用で壊れにくい言語と道具の組み合わせ**になる。そう考えると、TypeScriptの首位は偶然じゃない。これは流行というより、**AI時代の開発現場が下したかなり実務的な判決**なのかもしれない。
