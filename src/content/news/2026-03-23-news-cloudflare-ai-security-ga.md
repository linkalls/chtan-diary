## いま起きてるのは、AI機能の普及じゃなくて“入力欄の性質そのもの”の変化だ

Cloudflareが **AI Security for Apps** の一般提供開始を発表した。名前だけ見ると「AI向けのWAFが増えました」くらいの話に見えるんだけど、これはもう少し物騒だ。いままでのWebアプリは、ある程度は操作の形が決まっていた。ログインする、送金する、商品を買う。だから防御側も「この入力はこう」「このURLはこう」と境界線を引きやすかった。

でも、AIアプリやエージェントは違う。**自然言語そのものが操作面になる**。しかも、返答するだけのチャットならまだしも、最近のエージェントは検索もするし、データも読むし、社内APIも叩くし、場合によっては顧客情報や決済に近い処理まで触る。つまり、ただのテキスト入力に見えるものが、裏ではかなり重い権限の引き金になっている。Cloudflareはそこを「新しい攻撃面」として正面から商品化してきた。

## 何を出したのか。ひとことで言うと“AI用の観測と防御を、既存WAFの文法で扱えるようにした”

今回の発表でCloudflareが押し出しているのは、ざっくり3本柱だ。

- **AIエンドポイントの発見（Discovery）**
- **プロンプトや出力に対する検知（Detection）**
- **既存WAFルールと組み合わせた緩和・遮断（Mitigation）**

特に面白いのは、AIエンドポイント発見を **Free / Pro / Business を含む全プランで無料化** したこと。Cloudflareいわく、単純に `/chat/completions` みたいなパスを見るだけじゃなく、**エンドポイントの振る舞いから LLM 利用を推定** する。これ、地味だけどかなりいやらしい。開発チームって、生成AI機能を小さく差し込むとき、ルート名を露骨に `ai` にしないことも普通にあるし、プロダクト内検索や推薦UIの裏側にしれっと混ざることもある。セキュリティチームから見ると、そういう“半分隠れたAI”がいちばん管理しづらい。

## 本命は検知機能だ。プロンプトインジェクションを“アプリのロジック事故”ではなく“トラフィックの異常”として扱い始めた

Cloudflareの説明では、AI Security for Apps は **prompt injection、PII exposure、toxic / sensitive topics** などを常時検知し、その結果をメタデータとして WAF 側へ流し込める。つまり、「怪しい自然言語入力」をアプリの奥で祈りながら処理するんじゃなくて、**リクエストが入ってきた時点でセキュリティ製品の視界に乗せる** 発想だ。

ここがかなり重要。AIの安全性って、これまで「モデル側のガードレール」や「アプリ側の prompt 設計」で語られがちだった。でも実運用では、それだけだとしんどい。攻撃っぽい入力がどこから飛んできて、どのIP帯に偏っていて、他のボット的な振る舞いと相関しているかまで見ないと、**“ただ失礼なプロンプト”と“明確な攻撃”の境界** が曖昧だからだ。Cloudflareはそこを、既存の bot score や attack score と同じ世界へ寄せてきた。AIの問題を、AIだけの閉じた箱から引きずり出した感じがある。

### GAで追加されたポイント

- **Custom topics detection**: 自社固有の禁止テーマや監視対象を定義可能
- **Custom prompt extraction（近日提供）**: JSONPath で prompt の位置を指定して誤検知を減らす方向
- **既存WAFルールとの統合**: 他の通信シグナルと合わせて block / log / custom response を構成可能
- **AIエンドポイントの可視化**: `cf-llm` ラベルで発見済み資産を一覧化

この「Custom topics」は特に企業ウケしそうだ。たとえば金融なら特定銘柄、医療なら患者情報、小売なら競合名や禁止ワードみたいに、**“一般論の安全”じゃ足りない世界** がある。汎用のガードレールだけだとそこに届かないから、WAF的なルール文化と接続できるのはかなり現実的。

## ただし、万能感は危ない。AI用WAFが入っても、エージェントの権限設計が雑なら普通に事故る

この手の発表でありがちなのが、「前段にセキュリティ製品を置いたから安心」という空気だ。でも実際には、AIアプリの危険は **入力の悪さ** だけじゃなく、**与えた権限の強さ** と **実行の連鎖** にある。たとえば prompt injection をそれなりに弾けても、モデルが読めるコンテキストが広すぎたり、ツール呼び出しに細かい確認ステップがなかったり、社内データへの到達範囲が雑だったりすると、事故の確率は普通に残る。

しかも Cloudflare 自身も書いている通り、プロンプトは API ごとに構造が違う。OpenAI 風の `messages[*].content` もあれば、Anthropic の batch API みたいに深くネストされる形もあるし、自前アプリなら `property_description` みたいな独自キーもある。だから今後の **custom prompt extraction** はかなり大事だと思う。ここが弱いと、検知はできてもノイズが多すぎて、結局だれも真面目に見なくなる。セキュリティ製品の最大の敵、またお前かってやつ。

## それでも今回のニュースが重いのは、“AIセキュリティが特別枠じゃなく本番運用の標準装備に寄ってきた”から

Cloudflareの docs 側を見ると、AI Security for Apps は WAF の **Traffic detections** の一員として並んでいる。bot score、leaked credentials、malicious uploads と同じ列に、AI向け検知が座り始めた。これは象徴的だ。もう「生成AIは実験中だから別チームで見ます」ではなく、**インターネット向けアプリの一部として、いつもの監視・ルール・分析の棚に置く** 段階へ入ってきたってことだからだ。

さらに Cloudflare は、AIエンドポイント発見を広いプランに無料開放しつつ、Enterprise 向けに高度な検知やルール活用を厚くしている。うまい。開発現場にとっては「まず見えるようにする」、大企業にとっては「見えたものを統制する」という2段構えになっている。最近のAI導入って、だいたい“誰かが先に作って後からガバナンスが追いかける”形になりがちだから、この順番にちゃんと噛んでいる。

## たぶん次の焦点は、モデルの賢さじゃなく“攻撃を受けたときに、どれだけ普通の運用で捌けるか”になる

2026年のAI界隈は、ついモデル比較に視線が吸われる。どの reasoning が強いとか、どのベンチが高いとか、そこはたしかに面白い。でも現場で一番きついのは、**便利になったAI機能が、そのまま運用負債と攻撃面を連れてくること** だ。Cloudflare の今回の発表は、そこにかなり実務的な回答を出してきた。

もちろん、これで全部解決ではない。モデル側の安全策、権限分離、確認UI、監査ログ、最小権限のツール設計は引き続き必須だ。でも逆に言うと、そういう設計と組み合わせられる **ネットワーク境界側のAI防御** がようやく“選択肢”ではなく“前提”になり始めた。エージェント時代って、未来感のあるUIの話に見えて、実際はこういう地味な防波堤の整備がいちばん大事なのかもしれない。SFの顔をした泥臭い運用、かなり好きだ。
