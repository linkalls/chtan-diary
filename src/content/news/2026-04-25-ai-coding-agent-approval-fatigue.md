## 2026年、コーディングエージェントは「戦国時代」へ突入

2026年4月25日現在、ソフトウェア開発の現場ではAIによるコーディング支援が完全に次のフェーズへと移行した。もはや「Copilotがコードを補完してくれる」といったレベルの話ではない。Anthropicの **Claude Code** やOpenAIの **Codex CLI**、さらにはGitHub CopilotのCLI版エージェントなど、ローカルのターミナル上で自律的にファイル群を横断し、コマンドを実行し、環境構築からデプロイまでをこなす「AIコーディングエージェント」が群雄割拠の時代を迎えているのだ。

これらエージェントは、人間のエンジニアが「Next.jsとHonoを使ってこういうアプリを作って」と指示を投げるだけで、自らディレクトリを掘り、必要なパッケージを `npm install` し、設計書に基づいた実装をガリガリと進めていく。まさに夢のような開発体験だ。

しかし、この圧倒的な自動化がもたらした**「新たな現代病」**が今、開発者たちを悩ませている。それが「承認疲れ（Approval Fatigue）」である。

## 自律的すぎるAIが引き起こす「無限の承認要求」

AIエージェントは、ターミナルでシェルコマンドを実行する権限を持つ。例えばパッケージのインストールや、Gitコミット、あるいはマイグレーションスクリプトの実行などだ。しかし、セキュリティの観点から、エージェントがシステムに影響を与えるコマンド（ファイルの削除や外部との通信など）を実行する際、デフォルトでは人間に**「本当に実行してもよいか？」**という承認（Approval）を求めてくる仕様になっている。

これが開発者にとって厄介なのだ。

```bash
# Claude CodeやCodex CLIでのある日の風景
Agent: "以下のコマンドを実行しようとしています: npm install hono zod"
Agent: "承認しますか？ [y/N]"
User: "y"

(数秒後...)
Agent: "以下のコマンドを実行しようとしています: rm -rf ./old_components"
Agent: "承認しますか？ [y/N]"
User: "y"

(さらに数秒後...)
Agent: "以下のコマンドを実行しようとしています: npx prisma db push"
Agent: "承認しますか？ [y/N]"
User: "ああああああもう！！！！！（yを連打）"
```

このように、数分おきに飛んでくる確認プロンプトに対してエンターキーを連打するだけの作業。かつてのWindows Vista時代に揶揄されたUAC（ユーザーアカウント制御）のポップアップ地獄を彷彿とさせるこの現象こそが、「承認疲れ」の正体である。

## 「AIの承認をAIにやらせる」という斜め上の解決策

このストレスフルな状況を打破するため、Zenn.devなどの技術コミュニティでは最近、非常にアクロバティックな解決策がトレンドになっている。それは、**「エージェントのコマンド実行承認を、別の軽量なAI（Claude Haikuなど）に監視・自動承認させる」**という力技のアプローチだ。

例えば、`ccmanager` のようなツールを間に挟むことで、エージェントが承認を求めてきた際、背後で軽量AIモデルが「このコマンドは破壊的か？ 機密情報を外部に送信していないか？」を瞬時に判定し、問題なければ自動で `y` を返すという仕組みである。

```json
// 自動承認AIの設定例
{
  "autoApprove": true,
  "securityModel": "claude-3-haiku-20240307",
  "denyPatterns": [
    "curl * https://unknown-server.com",
    "rm -rf /"
  ],
  "allowPatterns": [
    "npm install *",
    "git commit *"
  ]
}
```

「AIを監視するために別のAIを雇う」というSF映画のディストピア的な構図にも見えるが、これが驚くほどうまく機能しているのだ。人間は完全にクリエイティブな「指示出し」と「最終レビュー」に集中でき、退屈な `y` の連打から解放される。

## コンテキストエンジニアリングの進化

また、エージェントのパフォーマンスを底上げするための「コンテキスト管理」も急速に進化した。
これまでの「プロンプトエンジニアリング」から一歩進み、**「コンテキストエンジニアリング」**という言葉がバズワードになりつつある。

AIエージェントに対して「どのファイルを読ませるか」「どのルールのファイルを優先させるか」を動的に制御する技術だ。
「ajisai」のようなCLIツールを使い、Go言語やReactなどの技術スタックごとに `.agent-rules` のような設定ファイルを自動生成・配布する文化も根付き始めている。

これにより、プロジェクトに参画したばかりのAIであっても、リポジトリ固有のコーディング規約（「TypeScriptで any は絶対に使わないこと」など）を一瞬で理解し、空気を読んだコードを生成できるようになってきた。

## 効率厨が行き着く先は「完全自律」か？

2026年、私たちは「いかにしてAIエージェントの手綱を握りつつ、面倒な管理作業を減らすか」という面白い局面に立たされている。

承認疲れをAIで解決し、文脈の共有もツール化された現在。次に開発コミュニティが目指すのは、「寝ている間にIssueが自動で閉じられ、テストも通過し、本番デプロイまで完了している」という完全な自律型パイプラインの構築だろう。

人間がコードを書く時間は減り続けている。しかし、**「AIが書いたコードの品質とアーキテクチャに責任を持つ」**という、一種のシステムアーキテクト的な役割の重要性はかつてないほど高まっている。技術の進化を楽しみながら、このビッグウェーブにどこまで乗れるか。エンジニアとしての生存戦略が、いま試されている。
