## なぜ今、Reactを捨てるのか？

Remixといえば、React Routerの作者たちが作り上げ、その後Shopifyに買収されて大きな話題を呼んだフルスタックWebフレームワークです。Reactの良さを活かしつつ、Web標準に寄り添った設計で熱狂的なファンを獲得してきました。

しかし、2026年現在のWeb開発トレンドは**「サーバーファースト」と「AIの統合」**に大きくシフトしています。Figmaの最近のレポートでも指摘されている通り、RSC（React Server Components）などの普及により、重い処理はサーバー側に寄せるのが当たり前になりました。

そんな中、Remix陣営は次のように判断したと推測されます。

- **AIによるコード生成・最適化の限界**: 既存のReactのエコシステムや仮想DOMの仕組みが、AIエージェントによる自動最適化のボトルネックになっていた。
- **パフォーマンスの壁**: Reactのライフサイクルに引きずられることなく、ネイティブに近い速度でDOM操作を行いたい。
- **真のWeb標準への回帰**: コンポーネント指向を維持しつつ、より軽量でブラウザネイティブなAPIへ移行する。

## Remix 3の「AIファースト」とはどういうことか？

「AIファースト」という言葉はバズワードのように聞こえますが、フロントエンドの文脈では非常に具体的な意味を持ちます。

1. **AIエージェントが理解しやすいAST（抽象構文木）**
2. **コンパイル時の自動コード分割と最適化**
3. **動的なUIのストリーミング生成**

Remix 3では、開発者が書いたコードをコンパイラ（おそらくRustやZigで書かれた超高速なもの）が解析し、**AIが実行時のコンテキストに合わせて最適なレンダリングパスを決定する**ような仕組みが導入されると噂されています。

```typescript
// 妄想コード：Remix 3の新しいコンポーネント定義？
import { createRoute, AIStream } from "@remix-run/core";

export default createRoute({
  loader: async ({ request }) => {
    // サーバーサイドでのデータ取得
    const data = await fetchSomeData();
    return data;
  },
  render: ({ data }) => (
    <section>
      <h1>超高速レンダリング</h1>
      {/* AIがユーザーの通信環境に合わせて静的HTMLかCSRかを自動フォールバック */}
      <AIStream data={data.heavyContent} fallback={<Spinner />} />
    </section>
  )
});
```

このように、従来のReactの `useState` や `useEffect` といったフックの概念から脱却し、より宣言的でインテリジェントなAPIになる可能性が高いです。
