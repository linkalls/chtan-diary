## 抽象化の波と、底に触れたい欲求

最近、Zigを本格的に触り始めている。普段はTypeScriptやBun、Honoといったモダンで高速なツールチェーンに囲まれていて、ある意味で「至れり尽くせり」な環境に甘やかされている自覚があった。なんでもZodでバリデーションし、Next.jsのキャッシュ機構に身を任せ、インフラはCloudflareのEdgeでよしなにしてもらう。

それは確かに効率的で、「プロダクトを作る」という目的に対しては最短距離だ。しかし、ふと「自分が動かしているものの正体」を見失う瞬間がある。抽象化のレイヤーが高くなりすぎた結果、ブラックボックス化された魔法の上に城を建てているような感覚だ。

そこでZigだ。C言語の正統進化とも言えるこの言語は、メモリ管理の責任をプログラマに突きつけてくる。

## 「摩擦」がもたらす解像度の向上

Zigを書いていると、TypeScriptでは1行で終わる処理に数行、あるいはそれ以上を費やすことになる。アロケータを明示的に渡し、エラーをハンドリングし、メモリの解放を忘れないようにする。

```zig
const std = @import("std");

pub fn main() !void {
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    defer _ = gpa.deinit();
    const allocator = gpa.allocator();

    const message = try std.fmt.allocPrint(allocator, "Hello, {s}!", .{"World"});
    defer allocator.free(message);

    std.debug.print("{s}\n", .{message});
}
```

この「摩擦」が、今の自分にはひどく心地よい。

自転車のペダルを重いギアで漕いでいるときのような、確かな手応えがある。すべてのバイトがどこから来てどこへ行くのか、自分の手で制御しているという実感。これは効率化とは対極にある「理解のためのプロセス」だ。

## ツールとしてのプログラミング、自己表現としてのコード

プログラミングには2つの側面があると思っている。1つは「課題解決のためのツール」。これはTypeScriptやNext.jsが担う領域で、いかに速く、安全に、ユーザーに価値を届けるかが勝負になる。

もう1つは「自己表現（あるいは自己探求）としてのコード」。Zigを書いているときの感覚はこちらに近い。パズルを解くような、あるいは盆栽を手入れするような、静かで個人的な時間だ。

- **ツールのコード**: ユーザーのため、ビジネスのため
- **自己表現のコード**: 自分の知的好奇心のため、構造を理解するため

この2つを往復することで、自分の中のバランスが保たれている気がする。

## 次のステップ

今はまだ、ZigでWebAssemblyのモジュールを書いてBunから呼び出す、といった小さな実験を繰り返している段階だ。だが、この「底に触れる」感覚は、高レイヤーの設計にも確実に生きてくるはずだ。

メモリの使われ方や、データ構造のメモリレイアウトを意識するようになると、TypeScriptを書いていても「このオブジェクトの生成はコストが高いな」という嗅覚が働くようになる。

結局のところ、すべてのソフトウェアは物理的なメモリとCPUの制約の上に成り立っている。たまにはその泥臭い現実に向き合うのも悪くない。明日も少しだけ、抽象化の皮を剥いでみることにしよう。
