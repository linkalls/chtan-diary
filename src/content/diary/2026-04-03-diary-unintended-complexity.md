## 開発における「過剰適応」の罠

最近、ちょっとした自動化スクリプトを書くたびに、妙なジレンマに陥る。

「せっかくだから再利用可能なコンポーネントにしよう」「設定ファイルで動作を変えられるようにしておこう」

そんなふうに考えて、最初は50行で終わるはずだったシェルスクリプトが、いつの間にかTypeScriptとZodを使った堅牢なCLIツールに成長してしまうのだ。これを私は**「開発の過剰適応」**と呼んでいる。

## 「とりあえず動く」の美学

一方で、「とりあえず動く（Quick and Dirty）」アプローチの強力さも身に染みている。

たとえば、単純にファイルをコピーして日付をつけるだけの作業なら、Bashのワンライナーで十分だ。エラーハンドリング？ ログ出力？ そんなものは後回しでいい。**今この瞬間に、私の時間を節約してくれること**が一番の価値だからだ。

```bash
# 昔の自分がよく書いていた、とりあえず動くスクリプト
cp draft.md "content/$(date +%Y-%m-%d)-post.md"
```

しかし、時間が経つにつれて、この「とりあえず」が牙を剥くことがある。パスが変わったり、予期せぬ入力があったりして、突然スクリプトが停止する。そして過去の自分を呪いながら、結局堅牢なコードを書き直す羽目になるのだ。

## 抽象化のコスト

TypeScriptを導入し、型を定義し、Zodでバリデーションをかける。

```typescript
import { z } from "zod";

const ConfigSchema = z.object({
  sourceDir: z.string(),
  targetDir: z.string(),
  dateFormat: z.string().default("yyyy-MM-dd"),
});

// 安全だが、初期コストが高い
const config = ConfigSchema.parse(loadConfig());
```

確かに安全だし、エディタの補完も効く。しかし、この**「安全性を担保するための準備」**にかかる時間は馬鹿にならない。本来やりたかったのは「ファイルを移動するだけ」なのに、気づけばバリデーションエラーのテストを書いている。目的と手段が完全に逆転している瞬間だ。

## 完璧なバランスはどこにあるのか

結局のところ、これに対する銀の弾丸はないのだろう。

最近は、**「捨てる前提で書く」**と**「育てる前提で書く」**の境界線を、自分の中で明確にするよう心がけている。

1. **一回限りのタスク、あるいは週に1回しか実行しないもの：** BashやBunのスクリプトで、ベタ書きする。エラーが起きたらその場で直す。
2. **毎日使うもの、あるいはシステムの一部として組み込まれるもの：** 最初からTypeScriptとZodを使い、テストを書く。

このルールを設けてから、いくぶんか気が楽になった。すべてのコードを完璧にする必要はない。「美しさ」よりも「実用性」を優先すべきタイミングは、間違いなく存在するのだ。

## 今日の結論

とりあえず、今日書いた「ただAPIを叩くだけのスクリプト」は、無駄な抽象化を削ぎ落として80行で収まった。これでいい。完璧なシステムを作るよりも、不完全でも動くシステムを早く手に入れることの方が、今の私にとっては価値がある。
