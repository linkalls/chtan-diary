## 効率と無駄のジレンマ

「効率厨」を自称していると、どうしても「すぐに役立つ知識」ばかりを追い求めてしまう傾向がある。TypeScriptの新しい型推論の仕組みや、Bunの最新アップデート、Edge Runtimeでのルーティング最適化など、学べば即座にコードの実行速度や開発体験の向上に直結するものたちだ。

しかし、最近はあえて「日本史」という、プログラミングの効率化とは対極にあるような分野に時間を割いている。

最初は受験という明確な目的のためだったが、次第にその過程自体に不思議な魅力を感じるようになってきた。

## なぜ丸暗記ではなく「構造」なのか

日本史の学習において、単なる年号や人名の丸暗記は苦痛でしかない。これは、プログラミングにおいて意味を理解せずにスニペットをコピペするのと似ている。

だからこそ、歴史の「構造」を理解することに注力している。

*   **誰が**：どの時代の、どの階層の人物か
*   **なぜ**：どのような政治的・経済的背景があってその行動を起こしたのか
*   **結果**：それが後世のシステム（法制度や文化）にどう影響したか

これらは、まさにシステムのアーキテクチャ設計や、技術の変遷（なぜReactから派生して様々なフレームワークが生まれたのか）をトレースする感覚に近い。

### 歴史をモデリングする

歴史的な出来事を頭の中でデータモデルとして構築してみる遊びが、意外と面白い。

```typescript
type HistoricalEvent = {
  era: string;
  actors: string[];
  catalyst: string;
  impact: string;
};

const taikaReform: HistoricalEvent = {
  era: "Asuka",
  actors: ["Naka no Oe no Oji", "Nakatomi no Kamatari"],
  catalyst: "Concentration of power by the Soga clan",
  impact: "Centralization of the state (Ritsuryo system foundation)"
};
```

このように構造化して捉えることで、「ただの昔の話」が「過去の人間たちが実装した社会システムのアップデート履歴」として読めるようになる。

## 無駄な知識がもたらす「遅延評価」

プログラミングの知識は、使えばすぐに結果が出る（即時評価）。一方で、歴史や哲学などの知識は、いつどこで役立つか分からない（遅延評価）。

一見無関係に見える歴史の知識が、ふとした瞬間にシステム設計のメタファーとして機能したり、人間の行動原理を予測するヒントになったりする。

効率だけを追い求めて一直線に進むのも悪くないが、たまにはこうして「無駄」と思われる知識の森に迷い込むことで、結果的により柔軟で強靭な思考回路が形成されるのかもしれない。

2026年3月19日の朝、今日もまた、古代の律令国家のバグ修正（法改正）の歴史を眺めながら、そんなことを考えている。
