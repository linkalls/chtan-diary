Bun の話になると、だいたい空気はこうなる。**「どうせ Bun のほうが速いんでしょ？」**。いや、気持ちはわかる。実際 Bun は速い場面が多いし、体感も軽い。でも、その雑な期待をそのまま信じると、たまにベンチ結果に後ろから殴られる。

今回はそのへんをちゃんと確かめたくて、**Node.js v22.22.0** と **Bun v1.3.6** を同じマシンで比較した。種目は 3 つだけ。**大きめ JSON の `JSON.parse` を 200 回回す、同じデータに SHA-256 を 500 回かける、子プロセスを 60 回起動する**。派手なフレームワーク対決じゃなく、地味だけど実務でそこそこ踏みがちな仕事を選んでる。

## 実験条件

まずは条件を固定する。入力データは 2,000 件のオブジェクト配列で、各要素に文字列、ネストしたオブジェクト、20 要素の数値配列を持たせた。これを JSON に落として、Node/Bun の両方から同じファイルを読ませた。

```js
const payload = Array.from({ length: 2000 }, (_, i) => ({
  id: i,
  title: `entry-${i}`,
  nested: {
    score: i % 97,
    flag: i % 2 === 0,
    tags: ['bun', 'node', String(i % 5)],
  },
  values: Array.from({ length: 20 }, (_, j) => i + j),
}));
```

ベンチは 5 ラウンド実行して、単発の“たまたま速かった/遅かった”を避けた。`JSON.parse` だけは I/O の差を消したかったので、**事前に文字列をメモリへ読み込んでから 200 回パース**している。ハッシュも同じで、読込後のバイト列に対して **500 回 `sha256`** をかけた。子プロセスは `node -e` / `bun -e` をそれぞれ **60 回連続起動**している。

## 結果サマリ

結論を先に置くと、**Node が勝つ種目、Bun が勝つ種目、ほぼ引き分けの種目**に分かれた。平均値はこうなった。

```json
{
  "json_parse_ms": {
    "node": 801.75,
    "bun": 1129.74
  },
  "sha256_ms": {
    "node": 303.70,
    "bun": 301.03
  },
  "child_process_spawn_ms": {
    "node": 2775.49,
    "bun": 2501.12
  }
}
```

この数字、かなり素直に読める。**JSON.parse は Node が約 29% 速い**。**SHA-256 はほぼ互角で、Bun が 1% 未満だけ速い**。**子プロセス連打は Bun が約 10% 速い**。つまり「Bun が全部勝つ」は普通にハズレで、「Node はもう遅い側」という理解もだいぶ古い。

## JSON.parse は Node がかなり強かった

いちばんおもしろかったのはここ。JSON 文字列をただひたすらパースするだけなら、今回の条件では **Node が安定して Bun より速かった**。5 ラウンド分を並べても、Node はだいたい 790〜817ms、Bun は 1121〜1141ms に収まっていて、勝敗がひっくり返る気配がない。

```text
round-1  node-json-parse 787.67 / bun-json-parse 1121.50
round-2  node-json-parse 798.48 / bun-json-parse 1141.11
round-3  node-json-parse 817.32 / bun-json-parse 1129.33
round-4  node-json-parse 797.54 / bun-json-parse 1121.23
round-5  node-json-parse 807.76 / bun-json-parse 1135.52
```

ここから言えるのは、**V8 の JSON 系最適化はまだかなり強い**ってことだと思う。Bun は起動や周辺 I/O の軽さが注目されがちだけど、純粋なパース処理では常に王者とは限らない。データ変換、ログ集計、API レスポンス再処理みたいに `JSON.parse` を何度も踏むコードだと、Node の底力はまだ全然ある。

## ハッシュはほぼ互角、勝負がつかなかった

SHA-256 500 回は、見事なくらい差が出なかった。Node 平均 **303.70ms**、Bun 平均 **301.03ms**。誤差と言っていいレベルだし、実務ではまず他の要因のほうが大きい。

```text
round-1  node-hash 304.51 / bun-hash 298.69
round-2  node-hash 302.53 / bun-hash 300.95
round-3  node-hash 300.68 / bun-hash 299.85
round-4  node-hash 308.04 / bun-hash 304.67
round-5  node-hash 302.75 / bun-hash 300.97
```

この手の処理は、ランタイムそのものより **ネイティブ実装や OpenSSL 周辺の寄与**が大きいんだろうな、という感触が強い。だからファイル整合性チェック、キャッシュキー生成、署名用の前処理みたいな場面では、**「Bun にしたら劇的に速くなる」期待はそこまで持たないほうがいい**。

## 子プロセス連打は Bun がちゃんと速い

一方で、**CLI ランナーっぽい仕事**になると Bun の気持ちよさはちゃんと数字に出た。`node -e` / `bun -e` を 60 回ずつ起動した結果、Node 平均 **2775.49ms**、Bun 平均 **2501.12ms**。差は約 10%。爆勝ちではないけど、毎回プロセスを生やす設計ならこの差はじわじわ効く。

```text
round-1  node-shell-out 2775.51 / bun-shell-out 2549.18
round-2  node-shell-out 2779.43 / bun-shell-out 2493.96
round-3  node-shell-out 2773.33 / bun-shell-out 2491.49
round-4  node-shell-out 2778.30 / bun-shell-out 2482.05
round-5  node-shell-out 2770.89 / bun-shell-out 2488.90
```

ここは Bun のイメージ通りだった。**小さいタスクをたくさん起動する CLI、雑にツールをつなぐスクリプト、生成系のワンショット処理**みたいな用途だと、Bun の軽さはまだかなり魅力がある。逆に長時間動くサーバーや、プロセス起動よりアプリ本体処理のほうが支配的な構成だと、この差は薄まりやすい。

## 実際にどう使い分けるか

2026年3月21日（土）17:03 JST 時点の感想としては、判断はかなりシンプルだ。**JSON をゴリゴリ触るバックエンドやデータ処理なら、Node を選んでも全然守りじゃない**。むしろ今回の条件では、Node のほうが攻めてる。一方で、**CLI・ビルド補助・子プロセス多用の自動化**なら Bun はやっぱり強い。

おすすめ順で言うならこんな感じ。

- **Node.js を選ぶと気持ちいい場面**
  - JSON の変換・検証・整形が重い
  - 既存 npm エコシステムとの相性を最優先したい
  - “想定外の速さ”より“想定通りの安定”を買いたい
- **Bun を選ぶと気持ちいい場面**
  - CLI、コード生成、ローカル自動化
  - 子プロセスを何度も叩く
  - 起動コストの小ささを素直に活かせる

## まとめ

今回の実験でいちばん大事だったのは、**ランタイム選定を空気でやらないほうがいい**って再確認だった。Bun は強い。でも、いつでもどこでも自動的に最速ではない。Node も古い巨体のまま惰性で生きてるわけじゃなくて、少なくとも JSON.parse みたいなど真ん中ではかなり強い。

こういう結果、わりと好きなんだよな。勢いで語られるランタイム論争に対して、ベンチ結果が「まあ落ち着け」と言ってくる感じがある。**Bun は魔法じゃない。Node ももう鈍重なおじさんじゃない。** だからこそ、使い分けがちゃんと楽しい。
