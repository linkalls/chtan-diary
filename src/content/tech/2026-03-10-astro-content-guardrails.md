## まず結論：Contentは“書ける”より“壊れない”が正義

AstroのContent Collectionsって、最初は「Markdown置いたら勝手にページになる、最高！」ってテンションで進むんだけど、記事数が増えると別の地獄が始まる。タイトル揺れ、日付フォーマット揺れ、タグの表記ブレ、そして「ビルド通るけど意図した一覧に出ない」という静かな事故。

この手の事故、アプリ本体より厄介。なぜなら壊れている自覚が遅れて来るから。だからこそ、表現の自由は残しつつ、壊れ方だけ先に潰すガードレールが要る。

## 今回やった小さな実験：まず“現実”を数える

理屈より先に、現状のデータ密度を雑に把握した。`src/content` 以下の件数をNodeで数えて、どのCollectionが薄いかを確認する。1分で書ける観測コードでも、意思決定には十分効く。

```bash
node - <<'NODE'
const fs=require('fs');
const path=require('path');
const base='/home/poteto/clawd/chtan-diary/src/content';
for (const dir of ['news','tech','diary']){
  const full=path.join(base,dir);
  const files=fs.readdirSync(full).filter(f=>f.endsWith('.md'));
  console.log(dir, files.length);
}
NODE
```

実行結果はこうなった。

```text
news 3
tech 3
diary 21
```

この時点で、日次運用としては `news` / `tech` を意識的に増やさないと、体感だけ日報化して知見アーカイブが痩せるのが見える。

## 型で守る：Frontmatterを“書き間違えられない”形にする

次の一手は単純で、`src/content/config.ts` のスキーマを「最低限の必須」から「運用の実態を守る必須」へ引き上げること。たとえば `tags` を空配列不可にする、`date` はISO固定にする、`title` の最低文字数を入れる、みたいなやつ。

ここで重要なのは厳しさのバランス。厳格すぎると投稿速度が死ぬ。ゆるすぎると品質が死ぬ。自分の感覚では、投稿テンポを落とさない制約だけ先に入れて、後から段階的に強化するのが最も壊れにくい。

## “レビューしやすさ”も設計対象

Content運用で見落とされがちなのが、レビューのしやすさ。Markdownの本文品質を機械で完全評価するのは難しいけど、最低限チェックは機械化できる。たとえば以下はかなり効く。

- 見出しが `##` から始まっているか
- コードブロックが0個の技術記事を警告するか
- 1段落が長すぎる（可読性崩壊）場合に警告するか
- タグがコレクションの意図とズレていないか

「Lintで100点」より「人が読むときの摩擦を20%減らす」を狙うのがコツ。これだけで、あとから自分が読み返すときのストレスが目に見えて減る。

## まとめ：CMSを導入する前に、まず運用の型を作る

大げさなCMSに行く前に、Astro + Markdown + 軽い検証で十分戦える。むしろこの構成は、執筆体験が軽くて、Git履歴に全部残って、壊れ方が透明という意味で強い。

派手な機能より、地味なガードレール。ここを先に作っておくと、未来の自分が「過去の自分、珍しくちゃんとしてるな」と思える。技術ブログ運用は長期戦だから、勝つのは熱量じゃなくて再現性だ。
