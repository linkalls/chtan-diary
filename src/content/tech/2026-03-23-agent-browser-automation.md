ヘッドレスブラウザの自動化といえば、これまでは Playwright や Puppeteer を使って、CSSセレクタやXPathを泥臭く指定してスクレイピングやE2Eテストを行うのが常識でした。しかし、AIエージェントの普及に伴い、その常識が大きく覆ろうとしています。

今回注目するのは、Rustベースの高速ヘッドレスブラウザ自動化CLIツール**『Agent Browser』**です。

これは単なるスクレイピングツールではなく、「AIエージェントがブラウザを操作・視認すること」を前提に設計された次世代のアーキテクチャを持っています。本記事では、このAgent Browserの何がすごいのか、そして実際の実行ログやコードスニペットを交えながら、そのポテンシャルを深掘りしていきます。

## なぜ従来のツールでは限界だったのか？

LLMにブラウザを操作させようとしたとき、最大のボトルネックになるのは「DOMの巨大さ」と「要素の特定」です。

通常のWebページは、LLMのコンテキストウィンドウにそのまま叩き込むにはノイズが多すぎます。大量の `div` タグ、不可視のスタイリング要素、動的に変化するクラス名……。これらをLLMに解釈させ、正確に `.click("#submit-btn-xyz")` を生成させるのは至難の業でした。

### Agent Browserの解決策：ARIAツリーの抽出

Agent Browserの最大の発明は、ページの構造を `snapshotFormat: "aria"` として、アクセシビリティツリー（ARIA）をベースに軽量化して抽出できる点です。

```json
// Snapshot抽出時のデータ構造イメージ
{
  "role": "button",
  "name": "ログイン",
  "targetId": "e12"
}
```

視覚的なノイズが削ぎ落とされ、LLMにとって「操作可能な要素」だけがID付きでリストアップされます。LLMは抽出された `targetId` を使って、`{"action": "act", "kind": "click", "targetId": "e12"}` と返すだけで、正確にボタンをクリックできるのです。

## 実際に動かしてみた：CLIからブラウザを制御する

では、実際にローカルでAgent Browserを立ち上げ、コマンドラインから操作してみましょう。

### 1. ブラウザの起動とページ遷移

まずはバックグラウンドでAgent Browserを起動し、特定のURLを開きます。

```bash
# Agent Browserの起動とページオープン
agent-browser open --url "https://example.com" --compact
```

実行すると、内部的にブラウザインスタンスが立ち上がり、ページのロードが完了するまで待機します。

### 2. ページ状態のスナップショット取得

次に、AIが解釈できる形でページの状態を取得します。

```bash
# ARIAベースのスナップショットを取得
agent-browser snapshot --refs "aria" --compact
```

**実行結果の抜粋:**

```text
[1] heading "Example Domain"
[2] paragraph "This domain is for use in illustrative examples..."
[3] link "More information..." (targetId: "e3")
```

このように、非常にクリーンな形でページ構造が返ってきます。Playwrightの生のHTMLダンプとは比較にならないほど、プロンプトのトークン消費を抑えられます。

### 3. 要素へのアクション（クリック）

スナップショットで取得したIDを使って、リンクをクリックしてみます。

```bash
# targetId "e3" をクリック
agent-browser act --kind click --targetId "e3"
```

ブラウザ側では正確にリンクがクリックされ、次のページへの遷移が開始されます。これら一連の動作が、すべてセキュアかつ高速なRustのランタイム上で処理されるため、オーバーヘッドが極めて小さいのも魅力です。

## LLM連携時のベストプラクティス

Agent Browserを自作のAIエージェントに組み込む際、いくつか気をつけるべきポイントが見えてきました。

1. **Waitの多用を避ける**
   Agent Browserはデフォルトで要素の可視性やロード状態を賢く待機します（Auto-waiting）。そのため、LLMに `act: wait` を無理に生成させるのではなく、必要なアクションを直接指示する方が安定します。
2. **`profile="chrome"` による既存セッションのハイジャック**
   Agent Browserは、単独の隔離されたブラウザだけでなく、ユーザーが普段使っているChromeのタブを拡張機能経由で直接操作するモード（`profile="chrome"`）を持っています。これにより、**ログイン済みのセッション**をそのままエージェントに引き継がせることが可能です。「Cookieをどうエージェントに渡すか」という面倒な問題が、これ一つで解決します。

## まとめ：UI自動化のパラダイムシフト

Agent Browserの登場により、「ブラウザ自動化」の主役は人間からAIへと本格的に移り変わりつつあります。

これまでSeleniumやPlaywrightのラッパーをこねくり回して作っていた不安定な自動化スクリプトは、近いうちに「スナップショットを取得し、LLMに判断させてクリックする」という堅牢なループに置き換わるでしょう。

特に、Next.jsやHonoなどでWebアプリを開発しているエンジニアにとって、この手のAgentベースのテストやスクレイピングは、今後の必須教養になるかもしれません。ローカルLLMと組み合わせることで、完全無料でセキュアな自律型リサーチボットを作ることも夢ではなくなりました。
