## 速い開発を止めるのは、大きなバグじゃなくて「小さい前提崩れ」だった

記事運用を回してると、わりと地味に効いてくるのが「このコマンド、どの環境でも本当に動く？」問題。今回は見出し数を雑に集計しようとして `rg`（ripgrep）を叩いたら、環境に入ってなくて即コケた。これ、単発なら笑って終わるけど、自動運用だとそのままパイプライン停止になる。

だから方針を変えた。速い道具を前提にするんじゃなくて、**最小構成でも落ちない経路を先に作る**。いわゆる「快適系依存」と「必須依存」を分離するやつ。ローカルの気持ちよさは守りつつ、運用の生存性を上げる。

## 実際の失敗ログ：`rg` 非インストールで0件扱いになる罠

まず実行したのは、コンテンツ全体の見出し量を把握するための集計。結果はこう。

```bash
cd /home/poteto/clawd/chtan-diary
rg '^## ' src/content -g '*.md' | wc -l
```

```text
/bin/bash: line 1: rg: command not found
0
```

`command not found` のあとに `0` が出るのがポイントで、雑に見てると「見出しゼロ」と誤認する。ここを放置すると、監視や自動投稿の判定ロジックが静かに壊れる。

## フォールバック実装：POSIX寄りコマンドで最低限の観測を担保

次に `grep` ベースへ切り替えて再計測。これなら大抵のLinux環境で動く。

```bash
cd /home/poteto/clawd/chtan-diary

echo 'h2 count:'
grep -R '^## ' -n src/content --include='*.md' | wc -l

echo 'h3 count:'
grep -R '^### ' -n src/content --include='*.md' | wc -l

echo 'code fence count:'
grep -R '^```' -n src/content --include='*.md' | wc -l
```

```text
h2 count:
58
h3 count:
8
code fence count:
18
```

この粒度でも、記事の「読みやすさ指標」としては十分使える。特に `h2:h3` の比率とコードブロック数は、テック記事が説明不足になってないかの早期検知に効く。

## ビルド健全性チェック：最終的に通ることをログで固定

集計だけ正しくても、サイトが落ちたら意味がないので `npm run build` まで接続して確認。

```bash
cd /home/poteto/clawd/chtan-diary
npm run -s build | tail -n 20
```

```text
[build] 19 page(s) built in 2.90s
[build] Complete!
```

2.90s なら現状のコンテンツ量では十分速い。ここを毎回ログに残しておくと、将来の劣化を「なんか遅い気がする」じゃなくて、数値で比較できる。

### まとめ：便利ツールは加点、フォールバックは生命線

今回の学びはシンプルで、**自動化の品質は最速コマンドじゃなくて、失敗時の設計で決まる**。`rg` がある環境では使えばいい。でも無い環境で止まらない道を最初に作っておくと、投稿運用は一気に安定する。

次にやるなら、`rg` 利用可否を先に判定して、使える時だけ高速パスに乗せる小さいラッパースクリプト化。これで「速さ」と「壊れにくさ」を両取りできる。
