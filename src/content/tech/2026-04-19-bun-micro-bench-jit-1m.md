## Bunの進化とJITの最適化速度

最近のBunの進化は目覚ましく、単なる「速いランタイム」から「エッジからCLIまで全てを担う基盤」へと変貌を遂げています。特にJIT（Just-In-Time）コンパイラのウォームアップ時間が短縮され、スクリプト実行時のレイテンシが極限まで削られています。

今回は、あえて非常に単純な「100万回ループの加算」という古典的なマイクロベンチマークを用いて、Bun 1.3環境での実効速度を計測してみました。

複雑なアプリケーションでのベンチマークも重要ですが、こうしたプリミティブな処理の速度は、AST解析や巨大な配列処理など、CPUバウンドな処理の基礎体力として直結します。

### 検証用のシンプルなスクリプト

検証に使ったコードは以下の通りです。余計なモジュールは読み込まず、純粋なループ処理のみを計測します。

```javascript
// benchmark.js
console.log('Hello, Bun v' + process.versions.bun);
const start = performance.now();
let sum = 0;
for(let i = 0; i < 1e6; i++) {
  sum += i;
}
console.log('1M loop took: ' + (performance.now() - start).toFixed(2) + 'ms');
```

このコードでは、`performance.now()` を用いてループの前後でミリ秒単位の時間差を求めています。

### 実際の実行結果（Bun 1.3.6）

手元のLinux環境（x64）で上記のスクリプトを実行した結果がこちらです。

```bash
$ bun benchmark.js
Hello, Bun v1.3.6
1M loop took: 10.95ms
```

100万回のループ処理がわずか **10.95ms** で完了しました。Node.js（V8エンジン）でも同様に高速ですが、Bun（JavaScriptCoreベース）は起動のオーバーヘッドがほぼゼロであるため、CLIツールとして単発で実行した際の体感速度が桁違いです。

### なぜここまで速いのか？

この速度の背景には、JavaScriptCore（JSC）の強力な最適化エンジンがあります。
具体的には以下のような要因が考えられます。

1. **Tiered JIT Compilation:** スクリプトの実行開始直後はインタプリタで高速に立ち上がり、ループなどのホットスポットが検出されると即座に最適化コンパイラ（DFG JIT / FTL JIT）に処理が引き継がれます。
2. **インラインキャッシュ:** 変数の型が固定されている（今回は常に数値）ため、型チェックのオーバーヘッドが省略されます。
3. **起動時の軽量さ:** V8に比べてランタイムの初期化が軽量であるため、プロセス起動からスクリプト実行までのラグが最小限に抑えられています。

### CLIツール開発における実務判断

このようなマイクロベンチマークは「実世界のWebサーバーの速度」とは必ずしも一致しません。しかし、**「CLIツール」や「ローカルエージェントのスクリプト」** を作る上では、この「起動の速さ＋即座のトップスピード」は決定的な武器になります。

例えば、ユーザーの入力に対して即座にファイル解析（ASTパースなど）を行い、結果を返すようなCLIを作る場合、Node.jsでは数十ミリ秒の「もたつき」を感じることがありますが、Bunではネイティブバイナリと同等のレスポンスを得られます。

今後は、Node.jsで書かれた既存の重いCLIスクリプトをBunに移行するだけで、開発体験（DX）が劇的に向上するケースがさらに増えていくでしょう。引き続き、より複雑なZodのバリデーションやSQLiteのクエリ処理など、実務に近いベンチマークも検証していきたいと考えています。
